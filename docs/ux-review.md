## 1. Persona 走查發現與處置

### P1 主要使用者：量化分析師（16:20 決策流）
| 發現 | 嚴重度 | 處置 |

| 切日期只能一天天點，無法一眼比較三天 | 中 | ✅ 新增三日概覽條（今/明/後 × 火燒雲區間 × 判定），點擊即切換 |

## 2. 同類 App 介面模式對照（採用／不採用與理由）

| App | 模式 | 決定 |
|---|---|---|
| **Alpenglow** | 單一大品質分數 + 極簡首屏 | ✅ 已同構（判定字 + 兩彙總區間）；**不採用**單點百分比——區間是本產品的紀律 |
| **SkyCandy / Sunsethue** | 未來多日品質預覽條 | ✅ 採用 → 三日概覽條（限 +2 天，守 3 天查詢紀律） |
| **PhotoPills** | 太陽方位地圖/AR 規劃 | ✅ 輕量採用 → 羅盤 SVG（AR/地圖 = 原生 Phase 3 候選） |
| **Apple Weather** | 白話一句摘要 + 時段條 | ✅ 採用 → plain summary；時間軸已有 |
| **Windy** | meteogram 多層雲剖面 | ❌ 不採用——資訊過載，違反「3 秒決策」原則；分層雲量以理由條列呈現 |
| **Google Maps 通勤** | 「現在出發 X:XX 抵達」 | ✅ 採用 → 建議出發時間（用建檔 access 路程，非即時交通） |
| 各天氣 app | 推播內嵌 app | ❌ PWA 不做 Web Push（iOS 不可靠）；推播維持 ntfy/Telegram，原生計畫處理 |

## 3. 已知取捨（記錄在案）

- 兩點位共用一次天氣呼叫（同城市誤差 ≪ 預報誤差；省 API 與電量）。
- 三日概覽用「當下最新預報」，後天欄位天然是低信心——條上判定僅供比較，
  點入後仍有 preliminary 虛線框與警語。
- demo 模式（`?demo=1`）固定「現在=16:20」且天氣依日期決定性變化，供展示與截圖重現。
